## 1. ホームと自分の言葉の連携 (RPA "見た目" ルーティンビルダー)

### ユーザー要望
- 「RPAみたいな見た目」の順次実行フロー UI (実装は普通の順序付きリスト)
- 1タップで読み上げを順次実行する「タスク連鎖」風の体感
- 言葉を追加すると同時にホームのセットにも追加できる
- 順番をスクロールで入れ替え可能
- 「まず読み上げる→次→次→...」のチェーン感

### 実装案 (RPA は見た目だけ、実装は単純な順序付き配列)
- 既存の `Affirmation` の `orderIndex` フィールドを活用 (新モデル不要)
- ホーム画面に「今日の読み上げセット」セクションを RPA フロー風に表示:
  - ステップ番号 (① ② ③ ...) + 矢印アイコン
  - 各ステップが言葉カード
- 編集モード: ドラッグで並び替え (`.onMove` または `.draggable`)
- 1タップで上から順に音読 (ReadAloudView の既存逐次再生を流用)
- 「言葉を追加」時に「ホームのセットに追加する」チェックボックス
- 完了したステップは ✓ マーク表示 (RPA タスク完了感)

### コスト
- 小〜中 (4-8時間) — モデル拡張は不要、ホーム再設計のみ

## 2. みんなの願い 24時間自動削除 + ロケーション最新100件のみ

### ユーザー要望
- 24時間で自動消える (現在は永続蓄積バグ)
- ロケーションごとに最新100件だけ表示
- ロケーション切替UI (国別タップで切替)

### 実装案
**A. クライアント側で表示制限 (Phase 2.0 — 即実装可)**
- `TimelineService.subscribePosts` を `.limit(to: 100)` + `expireAt > now` に
- ロケーション選択 Picker をタイムライン画面ヘッダーに追加
- AppStorage で選択ロケーション永続化

**B. サーバ側で物理削除 (Phase 2.1 — Blaze プラン要)**
- Cloud Functions の scheduled function で expireAt < now を batch delete
- Firestore TTL ポリシー (Blaze 必須) を ON

### コスト
- A: 小 (3時間)
- B: 中 (Blaze プラン必要、Cloud Functions 0.5日)

## 3. サクラ自動投稿 (運用無料ルール)

### ユーザー要望
- 1000テンプレを準備、定期で自動投稿
- 全言語対応
- リアル投稿があれば減らす、なければ100件埋める
- 自然に常時更新感

### 実装案
**運用無料ルール**
- Cloud Functions Scheduler (Free tier: 月125k 実行) を使う
- 1分毎ではなく15分毎に実行 → 1日96回
- Firestore 書き込み: 1日96投稿 × 30日 = 2,880 / 月 (Free 50k/日 内)
- 1000 テンプレを Firestore `templatePosts` コレクションに保存
- Cloud Function: 各 15分ごとに以下を実行
  - 各ロケーションの post 数を count
  - 100 件未満 & 直近24h 内のリアル投稿が 5件未満なら → サクラを 1-3件追加
  - リアル投稿が増えてきたら間引き

**テンプレ生成方針 (1000件)**
- カテゴリ別: 試験合格 / 大会優勝 / 健康 / 恋愛 / ダイエット / 仕事 / 家族 / 趣味
- 全言語 (ja/en/ko/zh-CN/zh-TW): 各200件 = 1000件相当
- "なんとかの試験に合格しますように" / "大会で優勝できますように" 等
- 神社・絵馬・短冊文化を調査して網羅
- リアルとサクラを混在: `isSakura: bool` フィールドでフラグ管理

### コスト
- テンプレ生成: 中 (1日 — LLM活用で自動生成可)
- Cloud Function: 中 (1日 — Blaze プラン要)
- 運用: 無料枠内で収まる

### 注意点
- サクラと明示しないが審査リスクあり (「ボット投稿は明示せよ」ガイドラインなし)
- アプリ内で「みんなの願い」は実ユーザーの言葉と説明、サクラはテンプレベースの願い文
- 投稿者UID として `_template_001` 等のプレフィックスでブロック対象から除外
- `isHidden`, `expireAt` も同様に管理し 24時間TTLで消える

## 5. 動作テスト計画 (Firebase 周り)

### テストシナリオ
| シナリオ | 期待動作 |
|---|---|
| ロケーションA から投稿 → ロケーションB から見る | B では見えない (ja_JP / en で分離) |
| 別アカウント / 別デバイスから見る | Firestore に記録され、両方で読み取り可 |
| 100件超えた時 | 古い順に削除 (Cloud Function) |
| ブロック後の見え方 | 該当UIDの全投稿が即時非表示 (locallyHidden) |

### バグチェックリスト (Firebase 周り)
- [ ] 匿名認証失敗時のフォールバック (オフライン)
- [ ] expireAt が既に過去の投稿が表示される (read-side filter 漏れ)
- [ ] 同UIDが複数デバイスで投稿した時の順序
- [ ] ロケーション切替後の再フェッチ
- [ ] サクラ生成失敗時のリトライロジック
- [ ] Firestore 接続エラー時の UI フォールバック

## 7. オンボーディング 朝夜トグル ↔ 設定 同期バグ修正

### バグ
- オンボで設定したトグルが Settings 画面で反映されない

### 原因
- オンボの @State 変数 (`morningEnabled` 等) が NotificationStep 完了時に @AppStorage に保存されていない
- Settings の @AppStorage キーと一致していない

### 修正
- `NotificationStep` の `onComplete` で AppStorage キー
  - `kotodama.notif.morningEnabled`
  - `kotodama.notif.morningHour`, `morningMinute`
  - 等を確実に書き込む

### コスト
- 小 (30分)
